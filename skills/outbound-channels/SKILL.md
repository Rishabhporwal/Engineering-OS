# Outbound Channels — WhatsApp + Transactional Email (Reference Patterns)

> **Reference implementation.** One concrete binding of the **outbound-notification seam** (see `engineering-os-blueprint/09-reference-architecture.md`). The OS is stack-agnostic — `STACK.md` may bind channels to the WhatsApp Cloud API (direct or via a BSP), SES/Resend/Postmark, SMS, or push. The *patterns* — consent + suppression checked on the send path, idempotent send, a unified channel abstraction, delivery tracking + retry/DLQ, and "the model phrases, never originates a number" — transfer; WhatsApp + email is the example.

The delivery surface for things like a **Morning Brief** (WhatsApp + email) when native apps are out of scope. **Owner:** Backend Engineer; Security/Compliance gates consent + suppression (a compliance P0); the AI/ML Engineer owns the *content* generation but **never the numbers in it**. Canon: `STACK.md`, `COMPLIANCE.md`.

## Invariants (NON-NEGOTIABLE)
1. **Check suppression + consent FIRST, synchronously on the send path** — not a nightly batch. SES auto-suppresses bounced/complained addresses; sending anyway tanks reputation and pauses the account. A send with no provable opt-in is a compliance violation, not a UX bug.
2. **Consent is channel-specific and externally collected.** WhatsApp **requires** explicit opt-in gathered via a *third-party* channel (website/SMS/IVR/in-person) — you may **not** collect WhatsApp opt-in inside WhatsApp, and email/SMS consent is **not** WhatsApp consent. Store who/when/where (`compliance-engine`).
3. **Idempotent send — no duplicate notifications.** Key on `(tenant, recipient, notification_event_id)`; write the dedup record **before** the side effect (or in the same transaction / outbox) so at-least-once redelivery never double-sends. (SES `SendEmail` is **not** idempotent — you own the key; Resend accepts a native `Idempotency-Key`.)
4. **The model phrases; it never originates a number.** Every figure in an AI-generated brief (revenue/%/count/currency) is **interpolated from the deterministic source of truth** (`metric-engine`), never generated by the model — an invented number is a correctness incident, not a wording nit.
5. **Respect the channel's rules or get throttled/blocked.** WhatsApp: marketing only with valid opt-in; outside the 24-hr window send **only an approved template**. Email bulk (>5k/day): pass SPF *and* DKIM with an **aligned** DMARC and **one-click unsubscribe** (RFC 8058). Separate marketing and transactional sending domains/IPs so a marketing complaint can't sink password-reset deliverability.

## WhatsApp Business (Cloud API)
- **24-hr customer-service window** opens from the user's last inbound message. **Inside it:** free-form/service messages are **FREE** (and unlimited since Nov 2024). **Outside it:** only an **approved template** (name + lang + params, values from the registry).
- **Pricing is now PER-MESSAGE** (the conversation-based model is deprecated — utility moved 1 Apr 2025, marketing + full cutover 1 Jul 2025). **Utility templates inside the open window → FREE**; **marketing + authentication → charged per delivered message** (marketing has no in-window freebie). A Click-to-WhatsApp ad / page-CTA opens a 72-hr free entry-point window. *(Exact per-country rates change frequently — re-check Meta's pricing page; treat any number as illustrative.)*
- **Watch these 2024–26 rules:** a per-user **marketing frequency cap** (since May 2024 — an accepted API call ≠ a delivered message); **no marketing templates to US (+1) numbers** (since Apr 2025); **silent utility→marketing re-categorization** (Meta's classifier can re-tag a live template with no notice → every later send bills at the marketing rate — **audit `pricing.category` on the status webhook**).
- **Templates** are pre-approved per category (marketing/utility/authentication); the #1 rejection cause is wrong category (a utility template that reads promotional). **Quality rating** (green/yellow/red over ~7 days from blocks/reports) gates the messaging tier (1K→10K→100K→unlimited).
- **Delivery tracking:** Meta POSTs `statuses[]` (`sent→delivered→read` or `failed`) to your webhook — **subscribe the WABA to your app** (the missing subscription is the classic silent-webhook bug). You can get both `delivered` and `failed` for one message (multi-device); `read` isn't emitted if the user never messaged you. **BSP vs direct:** Cloud API direct = free API + Meta's per-message charges; a BSP (Twilio/360dialog) adds onboarding/multichannel for a per-message markup (or a flat fee).

## Transactional email (SES / Resend)
```ts
await ses.sendEmail({
  FromEmailAddress: `no-reply@mail.${tenant.domain}`,   // dedicated transactional subdomain
  Destination: { ToAddresses: [to] },
  ConfigurationSetName: "transactional",                // → SNS/EventBridge events + suppression
  Content: { Template: { TemplateName, TemplateData: JSON.stringify(data) } },
});  // SES SendEmail is NOT idempotent — write your dedup key first.
```
- **DNS minimum (the 2024 Google/Yahoo + 2025 Microsoft bulk-sender rules):** bulk senders must pass **SPF *and* DKIM** and publish an **aligned** DMARC (≥ `p=none`), plus **RFC 8058 one-click unsubscribe** honored ≤2 days, with spam complaints <0.3% (aim <0.1%). DMARC needs *alignment*, not just a pass — start `p=none`, monitor `rua`, ramp to `quarantine`/`reject`. (Gmail escalated to **permanent rejection** Nov 2025; Microsoft enforces for >5k/day since May 2025.)
- **Reputation:** SES goes under review at >5% bounce, pauses at >10% bounce or >0.1% complaint. Handle **SNS bounce/complaint** events — Permanent bounces + all complaints → suppress immediately; Transient bounces are retryable (don't permanently suppress). Warm a dedicated IP over weeks (SES managed IPs auto-warm); the SES sandbox caps you at 200/day until you request production access.

## Unified notification abstraction
One provider-agnostic interface over WhatsApp/email/SMS/push: channel routing + fallback, **dedup once across all channels**, per-tenant from-domains + branding, per-tenant token-bucket rate limits, and **one place that enforces consent + suppression + idempotency**. Delivery is at-least-once + an idempotent consumer; permanent failures → DLQ, transient → retry with backoff (`stream-processing-consumers` discipline). A delivery-status webhook updates the notification's state; alarm on bounce/complaint/failed-rate.

## Effort-tier & cost note (`cost-routing-paradigms`)
Rendering + sending is deterministic. If a brief is AI-narrated, the LLM call is a tiered, cached `llm-gateway` call that receives the pre-computed numbers and only writes prose around them — it never computes or invents a figure. WhatsApp marketing and out-of-window template messages cost real money per send — gate them on consent + value, and prefer the free in-window/utility path where the product allows.

## Anti-patterns
Sending without a synchronous suppression/consent check · collecting WhatsApp opt-in inside WhatsApp (or reusing email consent for WhatsApp) · non-idempotent send (duplicate notifications on retry) · an LLM-invented number in a brief · free-form WhatsApp outside the 24-hr window (must be an approved template) · marketing template with no opt-in / to US +1 / ignoring the per-user frequency cap · not auditing `pricing.category` (silent utility→marketing re-bill) · weak/misaligned DMARC or no one-click unsubscribe (Gmail/Yahoo/Outlook rejection) · sharing one domain/IP for marketing + transactional · ignoring SNS bounce/complaint events (reputation tank → SES pause) · blasting a cold dedicated IP · no DLQ/retry on delivery failures · a notification path without per-tenant rate limits.
