# Engineering-OS improvement observations (from live monitoring)

> A running log of observations gathered by monitoring real Brain pipeline runs (scaffold → legacy-migration epic). Source of future improvement requirements for the Engineering OS itself. Not yet actioned unless noted. Started 2026-05-24.

## A. Improvement candidates (fix in the OS)

| # | Sev | Observation | Proposed improvement |
|---|---|---|---|
| O1 | **HIGH** | **Live secrets leaked into the `.engineering-os` audit trail.** When the Founder pasted dev credentials, agents wrote the *values* (Google OAuth `GOCSPX-…`, Shopify `shpss_…`, Gmail app-password, DB password) into **committed** files: `12-founder-decision.json`, `13-deploy-report.md`, `state/active.json`, `pending-founder-attention.md`, `live.log`. The repo has a GitHub remote. | Agents must **redact secrets before writing any artifact/journal/state/log** (pattern-match `sk-…`/`GOCSPX-`/`shpss_`/`shppat_`/`EAA…`/connection-string passwords → `***REDACTED***`). Add a redaction pass to the artifact-write path + a CI/secret-scan gate on `.engineering-os/`. *(Distinct from the Founder's deferred credential-rotation decision — the team writing secrets into committed files is a team-behavior defect regardless.)* |
| O2 | **MED** | **Monorepo scaffold did not gitignore `.env`.** `.env` is absent from the Brain repo `.gitignore` → a credential commit is one `git add` from happening. | The monorepo scaffold should ship `.env`/`.env.*` (minus `.env.example`) in `.gitignore` by **day one** as a non-negotiable. |
| O3 | **MED** | **No first-class way to resolve a mid-pipeline escalation.** A hard-gated requirement (e.g. the DPDP lawful-basis gate) had no `/decide` command — `/resume` doesn't satisfy a hard gate, so the Founder resolved it ad-hoc. | Add a `/decide <req> <ruling>` (or `/resolve-escalation`) command that records the Founder ruling + clears the named `build_gated_on`, then resumes. |
| O4 | LOW | **Usage rows log with lag** (sometimes minutes after the stage event), so live cost tracking is briefly imprecise. | Log the usage row synchronously at stage completion (immediately after the Agent result returns). |
| O5 | LOW | **Distribution friction:** plugin changes need version-bump → `/plugin update` → **session restart** to reach a running team (the running session holds the old plugin in memory; the version-keyed cache only refreshes on update). Non-obvious to operators. | Document the "restart to apply" rule prominently (ONBOARDING + a `/status`-surfaced plugin-version line); consider a session-start plugin-version check. |
| O6 | INFO | **Legacy-code-reading dominates migration-intake cost** (parent epic intake 80K, Child-1 intake 94K — driven by reading the legacy codebase, not context bloat). Current token levers don't touch this *variable* analysis cost. | Reinforces the **Graphify** adoption trigger (codebase-substance) — a queryable graph would cut the exploratory legacy reads. Revisit once the codebase has substance. |
| O7 | MED | **Parallel reviewers diverge on the *same* finding's severity/timing** (Child 1: inner-sync-writes — Shreya `defer` vs Tanvi `blocking`). Reconciliation resolved conservatively (good), but it cost a **bounce** that a shared bar could have pre-empted (Vikram would've fixed it on the first build pass). | Give Security + QA a **shared must-fix-now vs defer rubric** (esp. for rollout-ordering/“before-FORCE” classes), and surface known defer-vs-block calls to the *builder up front* so they're addressed in the first pass, not via a review bounce. |
| O9 | HIGH (latent) | **`.engineering-os` memory growth — bottleneck is specific, not general** (measured at 4.1M / 4 reqs). Context-cost is largely already designed-around (read-discipline + semantic index keep query cost flat as memory grows). The *real* risks: **(a) `state/active.json` is a hot-path file (read every action) at 60K/945 lines for 4 reqs with 3/4 terminal reqs unpruned + verbose payloads** — at ~236 lines/req this becomes a genuine token/latency bottleneck (~3MB read-per-action at 200 reqs); **(b) `state/*.bak` backups accumulate unbounded (~940K already)**; **(c) `runs/` git-bloat is linear (~197K/req)** → ~50MB at 200 reqs / ~250MB at 1000. | Apply Brain's own hot/cold tiering to its memory: **HOT** = slim `active.json` (in-flight only; prune terminal reqs → compact `registry.json` line + detail archived to the run folder), recent journal slices; **WARM** = decision-log + recent runs (grep/index only); **COLD** = old runs + summarized old journals + all `.bak` → S3/archive + manifest pointer; **RETRIEVAL** = semantic index stays the query layer. Priority: (1) prune `active.json` on completion [cheap, do soon — hot path]; (2) cap `.bak` (keep last N / gitignore); (3) tier old `runs/` to cold storage at a trigger (repo>50MB or >100 reqs); gitignore regenerated `dashboard.html`. |
| O8 | MED | **Bounce-fix runs skip full verification → regression.** Child 1's fix passed 37/37 tests but **skipped typecheck**, slipping a TS regression (`PrismaTx=never`) into re-review = a 2nd rework cycle. Fix runs are the *highest* regression-risk moment, yet got *less* verification. | The **v0.25.0 #3 no-SKIP principle already covers this** (a required check that didn't run ≠ pass) — get the team onto v0.25.0. Additionally: make **typecheck an always-on gate for any TS change** in the builder DoD, **explicitly applied to fix runs** (re-run the full acceptance contract after a bounce-fix, never a subset). |

## B. Already shipped — verify on the next clean run

**v0.26.0 — anti-rework (addresses O7 + O8; targets the ~36%/$9 rework on Child 1):**
- **System-prompt §11 "No avoidable rework"** (inherited by all builders): run the FULL verification set before every handoff (typecheck + tests + lint + acceptance contract; "tests pass" ≠ done); **bounce-fix re-runs the FULL contract, never a subset**; **shift-left self-review against the Security+QA gate criteria + the plan's `must-fix` items** before handoff. Reinforced concretely in `backend-developer` step 8. → kills the **O8** skipped-typecheck-regression class.
- **Architect folds known `must-fix` concerns into the builder's acceptance contract** (plan checklist) so they're built in pass 1, not bounced. → **O7** "not surfaced to builder" half.
- **Shared finding-severity rubric** (`docs/finding-severity-rubric.md`) — Security + QA classify must-fix-now-vs-defer with ONE rubric + conservative tie-break, so they converge instead of diverging on the same finding. → **O7** "gates diverge" half.
