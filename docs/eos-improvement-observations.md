# Engineering-OS improvement observations (from live monitoring)

> A running log of observations gathered by monitoring real pipeline runs (scaffold → legacy-migration epic). Source of future improvement requirements for the Engineering OS itself. Not yet actioned unless noted. This is an illustrative log — the *patterns* and the resulting OS fixes are what transfer; the specific examples came from one early adoption.

## A. Improvement candidates (fix in the OS)

| # | Sev | Observation | Proposed improvement |
|---|---|---|---|
| O1 | **HIGH** | **Live secrets leaked into the `.engineering-os` audit trail.** When the Stakeholder pasted dev credentials, agents wrote the *values* (OAuth client secrets, vendor API keys, an app password, a DB password) into **committed** files: a decision JSON, a deploy report, `state/active.json`, the pending-attention file, `live.log`. The repo has a remote. | Agents must **redact secrets before writing any artifact/journal/state/log** (pattern-match known secret formats → `***REDACTED***`). Add a redaction pass to the artifact-write path + a CI/secret-scan gate on `.engineering-os/`. *(Distinct from the Stakeholder's deferred credential-rotation decision — the team writing secrets into committed files is a team-behavior defect regardless.)* |
| O2 | **MED** | **Project scaffold did not gitignore `.env`.** `.env` absent from `.gitignore` → a credential commit is one `git add` from happening. | The scaffold should ship `.env`/`.env.*` (minus `.env.example`) in `.gitignore` by **day one** as a non-negotiable. |
| O3 | **MED** | **No first-class way to resolve a mid-pipeline escalation.** A hard-gated requirement (e.g. a compliance lawful-basis gate) had no `/decide` command — `/resume` doesn't satisfy a hard gate, so the Stakeholder resolved it ad-hoc. | Add a `/decide <req> <ruling>` (or `/resolve-escalation`) command that records the Stakeholder ruling + clears the named `build_gated_on`, then resumes. |
| O4 | LOW | **Usage rows log with lag** (sometimes minutes after the stage event), so live cost tracking is briefly imprecise. | Log the usage row synchronously at stage completion (immediately after the Agent result returns). |
| O5 | LOW | **Distribution friction:** plugin changes need version-bump → `/plugin update` → **session restart** to reach a running team (the running session holds the old plugin in memory; the version-keyed cache only refreshes on update). Non-obvious to operators. | Document the "restart to apply" rule prominently (ONBOARDING + a `/status`-surfaced plugin-version line); consider a session-start plugin-version check. |
| O6 | INFO | **Legacy-code-reading dominates migration-intake cost** (driven by reading the legacy codebase, not context bloat). Current token levers don't touch this *variable* analysis cost. | Reinforces the **Graphify** adoption trigger (codebase-substance) — a queryable graph would cut the exploratory legacy reads. Revisit once the codebase has substance. |
| O7 | MED | **Parallel reviewers diverge on the *same* finding's severity/timing** (Security `defer` vs QA `blocking` on the same finding). Reconciliation resolved conservatively (good), but it cost a **bounce** that a shared bar could have pre-empted (the builder would've fixed it on the first build pass). | Give Security + QA a **shared must-fix-now vs defer rubric** (esp. for rollout-ordering classes), and surface known defer-vs-block calls to the *builder up front* so they're addressed in the first pass, not via a review bounce. |
| O14 | **HIGH** | **`usage.jsonl` cost logging silently stopped for recent reqs** — two requirements completed with **0 usage rows** (logging worked for the prior post-restart reqs, then stopped). Combined with O13 (stale live.log) + O10 (no breakdown split), **the cost telemetry the whole optimization effort depends on is now unreliable** — recent runs are unmeasurable. | Make usage logging **robust + required per spawn** (not session/driver-dependent); add a session-start + per-stage assertion that the usage row was written (a missing row is a defect, like the no-SKIP rule); reconcile against run-folder artifacts as a backstop. *The cost-optimization program is blind without this.* |
| O13 | MED | **`live.log` narration silently stopped** — last entry stale while 3 more requirements completed (state + usage kept updating). Anyone relying on `/watch` would be blind to real progress, with no error — silence looks identical to "idle." | Make the live-narration write a **hard step** (or a heartbeat) so a dead narration is detectable; `/watch` should warn when `live.log` mtime lags `active.json`/`usage.jsonl`. Monitoring should key off **authoritative state + append-only usage**, not just `live.log`. |
| O12 | **HIGH** | **Cost is dominated by top-tier-model review ROUNDS, not builds — and any single bounce re-runs the FULL review.** Measured: a tuned run cut tokens ~30% vs a bounced earlier run (fewer builds + 1 persona not 2) but **cost stayed ~flat** because (a) one bounce still triggered a full second review round (same number of top-tier reviewer runs as the 2-bounce run), and (b) cost is concentrated in the top-tier stages (intake/architect/security×rounds/final), not the mid-tier builds the anti-rework fixes trimmed. *Reducing bounce COUNT barely moves $ when each bounce re-runs the whole review.* | **Scoped "delta re-review":** after a bounce-fix, Security+QA re-verify **only the bounced findings + a regression/diff check**, not a full from-scratch re-review (~halves the 2nd-round cost — the real bounce-cost lever). Also consider: a cheaper model for routine re-verification of an already-passed surface; and watch whether top-tier security depth can be tiered like personas were. |
| O11 | **HIGH** | **Verification VALIDITY gap — "ran + passed" ≠ "valid."** A run reached review with passing tests + a clean typecheck (the checks *ran*), but QA VETO'd: the integration tests ran under a **bypassed isolation context** (row-level isolation never actually exercised) and a security probe's contextless arm was **inert**. A green test under security-bypass / an inert probe is **false confidence — worse than no test.** Ensuring verification *runs* does not ensure it's *valid*. **RECURRING — confirmed across multiple runs:** inert probes; a tautological parity test ("tautology"); duplicate divergent fixture trees. Verification-validity defects are the **#1 recurring bounce cause** → strongest case to SHIP a proactive negative-control requirement. | For any security/tenancy/auth feature, **tests MUST run under the real (non-bypass) security context** (no privileged/isolation-bypass role), and every probe/test must carry a **negative control** — it must *fail when the protection is removed* (proves it actually exercises the path). Make "your verification must be able to fail" an explicit builder-DoD + QA-criteria item. |
| O10 | MED | **#1 token-breakdown logging degrades to `total_tokens`-only — the input/cache/output split isn't available to log.** Likely cause: the **subagent-spawn usage the orchestrator receives is total-only** (the harness doesn't surface the split), so the "omit-if-unavailable, never fabricate" rule correctly omitted them. **Consequence: per-stage context-vs-reasoning is still NOT measurable via `usage.jsonl`.** | If the split is genuinely unavailable: (a) the only reliable per-stage measurement is an **identical-task A/B** (run the same trivial req on two versions); (b) revisit if the Agent/Task result ever exposes `usage.input_tokens`/`cache_*`; (c) consider a cheap proxy — log the *artifact* + loaded-context byte sizes the agent reports, as a stand-in for context-load. Don't over-invest until the harness exposes the data. |
| O9 | HIGH (latent) | **`.engineering-os` memory growth — bottleneck is specific, not general.** Context-cost is largely already designed-around (read-discipline + semantic index keep query cost flat as memory grows). The *real* risks: **(a) `state/active.json` is a hot-path file (read every action) that grows with unpruned terminal reqs + verbose payloads** — at scale this becomes a genuine token/latency bottleneck; **(b) `state/*.bak` backups accumulate unbounded**; **(c) `runs/` git-bloat is linear per req** → tens of MB at hundreds of reqs. | Apply hot/cold tiering to the OS's own memory: **HOT** = slim `active.json` (in-flight only; prune terminal reqs → compact `registry.json` line + detail archived to the run folder), recent journal slices; **WARM** = audit log + recent runs (grep/index only); **COLD** = old runs + summarized old journals + all `.bak` → archive + manifest pointer; **RETRIEVAL** = semantic index stays the query layer. Priority: (1) prune `active.json` on completion [cheap, do soon — hot path]; (2) cap `.bak` (keep last N / gitignore); (3) tier old `runs/` to cold storage at a trigger (repo>50MB or >100 reqs); gitignore regenerated `dashboard.html`. |
| O8 | MED | **Bounce-fix runs skip full verification → regression.** A fix passed its tests but **skipped typecheck**, slipping a type regression into re-review = a 2nd rework cycle. Fix runs are the *highest* regression-risk moment, yet got *less* verification. | The **no-SKIP principle already covers this** (a required check that didn't run ≠ pass). Additionally: make **typecheck an always-on gate for any typed-language change** in the builder DoD, **explicitly applied to fix runs** (re-run the full acceptance contract after a bounce-fix, never a subset). |

## B. Already shipped — verify on the next clean run

**Anti-rework (addresses O7 + O8):**
- **System-prompt "No avoidable rework"** (inherited by all builders): run the FULL verification set before every handoff (typecheck + tests + lint + acceptance contract; "tests pass" ≠ done); **bounce-fix re-runs the FULL contract, never a subset**; **shift-left self-review against the Security+QA gate criteria + the plan's `must-fix` items** before handoff. Reinforced concretely in `backend-developer`. → kills the **O8** skipped-typecheck-regression class.
- **Architect folds known `must-fix` concerns into the builder's acceptance contract** (plan checklist) so they're built in pass 1, not bounced. → **O7** "not surfaced to builder" half.
- **Shared finding-severity rubric** (`docs/finding-severity-rubric.md`) — Security + QA classify must-fix-now-vs-defer with ONE rubric + conservative tie-break, so they converge instead of diverging on the same finding. → **O7** "gates diverge" half.

**Earlier:**
- **Token-breakdown logging** (input/output/cache split) — makes per-stage savings measurable.
- **Spike/planning lane** depth-modifier — right-sizes Security+QA depth on no-code artifacts (origin: a no-code spike spent ~40% of its cost on review).
- **Persona model-tiering** (small/large) — origin: two large-model personas on a spike where a bounded persona ran fine on a small model.

## C. What's working — keep, don't regress
- **Anti-blind-agreement** fired hard: an epic-level **challenge-back + decompose** (avoided a multi-hundred-K full-epic run) — the single biggest cost avoidance observed.
- **Personas earn their spend:** caught real criticals — an isolation-context leak through a connection-pool, a partial-isolation-deploy compliance window, a bad dependency pin, a data-residency gap.
- **Compliance hard-gate held against an explicit `/resume`** — no path let the team process live PII without a recorded basis. Compliance-as-P0 working.
- **Ground-truth rigor:** a stray backup file revealed a prior wrong data-residency region; a multi-row tenancy audit caught isolation-key gaps.
- **Gates do real independent re-verification:** the Engineering Advisor re-ran the codegen at final review; QA independently caught a codegen-pin bug.
