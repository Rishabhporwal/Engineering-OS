| # | Sev | Observation | Proposed improvement |

| O11 | **HIGH** | **Verification VALIDITY gap — "ran + passed" ≠ "valid."** v0.26 run reached review with 155 passing tests + tsc clean (Fix A worked — checks *ran*), but Tanvi VETO'd: the integration tests run as a **`BYPASSRLS` superuser** (RLS never actually exercised) and the `CF-SEC-1` probe's contextless arm is **inert**. Make "your verification must be able to fail" an explicit builder-DoD + QA-criteria item. |
| O10 | MED | **#1 token-breakdown logging degrades to `total_tokens`-only — the input/cache/output split isn't available to log.** First v0.26.0 run: intake row had no breakdown fields despite the cache being confirmed v0.26.0. | O9 | HIGH (latent) | **`.engineering-os` memory growth — bottleneck is specific, not general** (measured at 4.1M / 4 reqs). Context-cost is largely already designed-around (read-discipline + semantic index keep query cost flat as memory grows). The *real* risks: **(a) `state/active.json` is a hot-path file (read every action) at 60K/945 lines for 4 reqs with 3/4 terminal reqs unpruned + verbose payloads** — at ~236 lines/req this becomes a genuine token/latency bottleneck (~3MB read-per-action at 200 reqs); **(b) `state/*.bak` backups accumulate unbounded (~940K already)**; **(c) `runs/` git-bloat is linear (~197K/req)** → ~50MB at 200 reqs / ~250MB at 1000. | Apply Brain's own hot/cold tiering to its memory: **HOT** = slim `active.json` (in-flight only; prune terminal reqs → compact `registry.json` line + detail archived to the run folder), recent journal slices; **WARM** = decision-log + recent runs (grep/index only); **COLD** = old runs + summarized old journals + all `.bak` → S3/archive + manifest pointer; **RETRIEVAL** = semantic index stays the query layer. Priority: (1) prune `active.json` on completion [cheap, do soon — hot path]; (2) cap `.bak` (keep last N / gitignore); (3) tier old `runs/` to cold storage at a trigger (repo>50MB or >100 reqs); gitignore regenerated `dashboard.html`. |
