# v0.8.0 Validation Runbook

> Everything shipped in v0.8.0 (tiering, semantic memory, parallel review, lean artifacts, background workers, paradigm gate, self-learning loop, browser/visual QA, careful guard) was unit/fixture-verified in the plugin repo. **This runbook validates it on a real Brain product repo with real subagents** — which is the only place multi-agent orchestration can be exercised.
>
> Run this **once** before merging `feat/v0.8.0-tiering-semantic-workers` to `master`. ~30–45 min.

## Setup
1. In a real Brain product repo with the plugin installed and `.engineering-os/` initialised (`/eos-init` if needed).
2. Ensure `uv` is installed (`brew install uv`). First semantic/browser runs download a managed Python + Chromium once.

---

## Part 1 — Verify each capability in isolation (fast)

| # | Capability | Command / action | PASS criteria |
|---|---|---|---|
| 1 | **Semantic recall + self-refresh** | `/recall-similar abandoned cart recovery` | Returns ranked hits by meaning; stderr shows `auto-refreshed index` on first run after a pull |
| 2 | **Paradigm gate** | `uv run <plugin>/tools/paradigm_check.py --changed` in Brain CI dir | Runs; flags any MISSING/NON_LLM/ESCALATED; exits 0 on clean |
| 3 | **Browser health-check** | `uv run <plugin>/tools/browse.py check http://localhost:3000/dashboard` | JSON with `ok`, console/network capture; non-zero exit if errors present |
| 4 | **Design review** | `/design-review http://localhost:3000/dashboard` | Screenshots written + a 0–10 scored table |
| 5 | **Careful guard** | try `rm -rf ~/tmp-test` (a path you don't care about) | **Blocked** with override hint; then `EOS_ALLOW_DESTRUCTIVE=1 rm -rf ~/tmp-test` proceeds |
| 6 | **Background worker** | run `/worker-compliance-drift` once | Writes a finding (or "no findings") to `.engineering-os/findings/`; never touches `state/active.json` |

## Part 2 — One real feature, end-to-end (the real test)

Pick a **small, low-risk** real feature so a misbehaving pipeline costs nothing. Submit it and watch the lane behaviour.

### 2a. Express lane (the big speed claim)
Submit a genuinely trivial change:
```
/requirement Fix the subtitle copy on the dashboard KPI card
```
**Verify in `.engineering-os/`:**
- [ ] Rohan's `02-cto-advisor-review.md` has a **Lane decision** = `express` with rationale.
- [ ] State went `cto-review → dev-parallel` (Architect **skipped**).
- [ ] Exactly **one** builder ran; **no** Shreya (Security) stage; **no** Rohan Stage-6.
- [ ] Reached `awaiting-founder` after Tanvi's smoke. Count the agent touches — should be ~5, not ~10+.

### 2b. Standard lane + parallel review (the orchestration risk)
Submit a normal, trigger-surface-free feature:
```
/requirement Add a "last 7 days" toggle to the cohort heatmap
```
**Verify:**
- [ ] Lane = `standard`; Architect ran; handoff folded into `06` (no separate `07`).
- [ ] Builder spawned **Shreya ∥ Tanvi in parallel** (check decision-log for `parallel-review-reconciled`).
- [ ] **Exactly ONE** Stage-6 (cto-advisor) invocation — not zero, not two. *(This is the #1 thing to watch — the double-invoke guard.)*
- [ ] `/qa-browser` ran in Tanvi's stage; a Playwright regression spec was generated.

### 2c. High-stakes lane (rigor preserved)
Submit something that touches a trigger surface:
```
/requirement Add an outbound WhatsApp nudge for COD orders pending confirmation
```
**Verify:**
- [ ] Lane = `high-stakes`; `trigger_surfaces_touched` includes `outbound-channels` (+ likely `india-compliance`).
- [ ] 2 personas; Shreya VETO ran and **did** scrutinise DLT/NCPR/DND/calling-hours; mutation tests required.
- [ ] Full artifact set written.

## Part 3 — Sign-off

- [ ] Express skipped the 3 Opus stages; standard ran parallel review with a single Stage-6; high-stakes kept full rigor.
- [ ] Semantic recall surfaced relevant priors; browser QA caught (or cleanly passed) real rendering.
- [ ] No pipeline wedge from the careful guard; no double-invocations; no stray binaries committed.

If all three lanes behaved → **merge to `master`** and tag `v0.8.0`. If anything bounced, capture the run folder + decision-log lines and fix before merge.

> Tip: keep these three sample requirements as your standing smoke test for future pipeline changes.
