# 07 — Operations & Reliability

> What happens from "approved" through "running reliably in production for years." Deployment and
> release management, reliability engineering and SLOs, incident management, production support, and
> the operational artifacts (runbooks, postmortems) that keep the system alive and improving.

---

## 1. Deployment strategy

**Intent.** Release is decoupled from deploy; a bad change is caught automatically and reverses
itself before it harms many users.

- **Progressive delivery by default.** Change rolls out incrementally (canary → wider → full), not
  big-bang. The blast radius of a bad deploy is bounded by construction.
- **Feature flags decouple release from deploy.** Code can ship dark and be enabled separately,
  per-cohort/tenant, with a **kill switch** that disables a feature in seconds without a redeploy.
- **A monitored bake window.** After each rollout step, the change *bakes* under live traffic while
  the platform watches the SLOs and key signals.
- **Auto-rollback on breach.** If a signal crosses its threshold during the bake window, the rollout
  **reverses automatically** — recovery does not wait for a human to notice.
- **Immutable, promoted artifacts.** The exact artifact verified in lower environments is the one
  promoted to production; environments differ in config, not in build.
- **Reversible by design.** Every deploy has a tested rollback path; database migrations are staged
  so the previous version still runs.

The strategy's parameters (bake window length, rollback thresholds, canary steps) live in the
Canon's **deploy playbook**, aligned to the product's SLOs.

## 2. Release management

**Intent.** Every change to production is deliberate, recorded, and traceable to its requirement.

- **Change control.** A change reaches deploy only through the pipeline's gates (Stage 8 after the
  stakeholder gate). No out-of-band production changes.
- **Commit-boundary discipline.** Product code and the `.engineering-os/` audit trail are committed
  as **separate, explicit-path** stages — never blanket-staged. History is **append-only** (no
  rewrite). A stage is "done" only after the **push succeeds** (push-success gate).
- **Release notes** for every shipped requirement, generated by the Delivery Coordinator and linked
  to the requirement id and the decision log.
- **Tracker synchronization** keeps external project trackers consistent with reality, opt-in per the
  Canon; if no tracker is configured, the record still lands in the repo memory.
- **Release channels respect platform rules** — e.g. mobile over-the-air updates vs. native-binary
  releases follow the store's policy ([02 — Mobile Engineer](02-engineering-roles.md)).

## 3. Reliability engineering & SLOs

**Intent.** Reliability is an explicit, measured target with a budget — not an aspiration.

- **SLIs / SLOs / error budgets.** Each user-facing capability has Service Level *Indicators*
  (latency, availability, correctness, freshness), *Objectives* (the target), and an **error budget**
  (the allowed shortfall). Burning the budget changes behavior — it slows risky releases and
  prioritizes reliability work.
- **Resilience patterns are standard** — timeouts, retries with backoff and jitter, circuit
  breakers, bulkheads, back-pressure, and graceful degradation (the system sheds non-essential load
  rather than collapsing).
- **Failure is tested, not assumed** — failure injection / chaos exercises validate that the
  resilience patterns actually work before production proves they don't.
- **Capacity is planned** — load and performance testing against the budgets in
  [05 §7](05-engineering-standards.md); autoscaling is bounded; headroom is deliberate.
- **Toil is engineered away** — repetitive manual operations are automated; the toil ratio is a
  tracked metric to *reduce* ([06](06-quality-gates-and-metrics.md)).

## 4. Incident management

**Intent.** When something breaks, response is fast, coordinated, and learning-producing — and never
about blame.

### Severity ladder

A fixed severity scale (e.g. SEV1 = broad outage / data risk → SEV4 = minor, no user impact)
determines response urgency, who is paged, and communication cadence. The ladder is defined in the
Canon; the *structure* is fixed by the OS.

### Roles during an incident

| Role | Responsibility |
|---|---|
| **Incident Commander** | Owns the response; coordinates; decides; the single point of authority. Usually Platform/SRE. |
| **Operations / responders** | Diagnose and mitigate under the Commander's direction. |
| **Communications** | Keeps stakeholders informed at the cadence the severity demands. |
| **Scribe** | Records the timeline as it happens (feeds the postmortem). |

### Flow: detect → mitigate → resolve

1. **Detect** — automated alerting on SLO breach / error-budget burn / health signal; the
   correlation identity makes "what is happening" answerable fast.
2. **Mitigate first.** Stop the bleeding — kill switch, rollback, failover, scale, shed load — *before*
   root-causing. **Fast kill switches** (seconds, not deploys) exist for the highest-risk surfaces.
3. **Resolve** — restore normal service; confirm via the same signals that detected the incident.
4. **Learn** — a **blameless postmortem** (§6) within a fixed window; action items feed back into
   lessons-learned and the backlog.

## 5. Production support

**Intent.** The running system has a defined support model so problems are handled by design, not
heroics.

- **On-call** with a humane rotation, clear escalation paths, and a defined acknowledge/response SLA.
- **Runbooks** for every known failure mode and routine operation (§6) — on-call follows the runbook,
  not tribal memory.
- **Tiered support / triage** — a structured path from first report to the right responder, with the
  correlation id threading the report to a trace.
- **Operational readiness is a gate** — before a service goes live it passes a readiness checklist:
  health/readiness probes, real-network smoke, env-var validation, resource limits, observability
  wired, rollback rehearsed. Un-ready services do not get on-call.
- **Feedback loop to engineering** — recurring support load is a signal that routes to the backlog as
  a reliability or DX requirement, not absorbed indefinitely as toil.

## 6. Operational artifacts

The OS standardizes the documents operations runs on. Templates live with the Canon.

| Artifact | Purpose | Owner |
|---|---|---|
| **Deploy playbook** | The exact rollout/bake/rollback procedure + thresholds. | Platform / SRE |
| **Runbook** | Step-by-step for a specific operation or failure mode, with the signals to check and the actions to take. | Owning function |
| **Threat model** | The trust boundaries, threats, and mitigations for a system. | Security |
| **Postmortem** | Blameless incident analysis: timeline, contributing factors, what worked, action items with owners. | Incident Commander |
| **Incident runbook** | The severity ladder, paging, and the IC's decision tree. | Platform / SRE |

**Blameless postmortem discipline:** the analysis targets the *system and its conditions*, never the
people. A human error is treated as a signal that the system permitted the error — the fix is a
guardrail, not blame. Action items have owners and land in the backlog; the lesson lands in
`lessons-learned` so the organization stops repeating it.

---

## 7. The operations feedback loop

```
 Incident / support load / SLO burn / metric drift
        │
        ▼
 Postmortem / triage  ──▶  lessons-learned  ──▶  feeds intake & escalation judgment
        │
        ▼
 Action items  ──▶  backlog  ──▶  new requirements  ──▶  pipeline  ──▶  stronger gates / guardrails
```

Operations is not a dead end where problems are merely survived. Every incident, every recurring
support burden, and every SLO breach is an **input to engineering** — it becomes a requirement that
hardens the system through the same pipeline that built it. This loop is how the system gets *more*
reliable over time instead of decaying.
